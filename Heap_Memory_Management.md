# 배경 개념

- **세그먼트(segment)**: 주소 공간에서 특정 역할을 맡는 영역이다. 대표적으로 스택, 힙, 데이터, 코드 영역이 있다.
- **힙 영역(heap segment)**: 실행 중 필요한 메모리를 동적으로 할당받는 영역이다.
- **힙 할당기(heap allocator)**: 힙 영역의 메모리를 할당하고 해제하는 프로그램이다.
  - 필요한 이유: 메모리 할당과 해제 과정을 추상화하고 표준화해, 여러 환경과 운영체제에서도 일관된 방식으로 동작하게 해준다.

# 힙 메모리 관리(Heap Memory Management)

동적 메모리 할당은 실행 중에 변하는 메모리 수요를 처리하기 위한 방식이다. 필요한 만큼 메모리를 할당하고, 더 이상 필요하지 않은 메모리를 회수해 제한된 힙 공간을 효율적으로 사용하는 것이 목적이다.

1. **블록 표현(Representation)**: 힙 메모리를 어떤 단위의 블록으로 나누고, 각 블록에 어떤 정보를 저장할 것인가
2. **빈 공간 추적(Tracking)**: 사용 가능한 가용 블록을 어떻게 관리할 것인가
3. **할당/해제 정책(Policy)**: 어떤 블록을 선택하고, 어떻게 분할하고, 언제 병합할 것인가

```text
힙 메모리 관리
├─ 1. 블록 표현
│  ├─ 블록 구조
│  └─ 메타데이터
├─ 2. 빈 공간 추적
│  ├─ 암시적 가용 리스트
│  └─ 명시적 가용 리스트
├─ 3. 할당/해제 정책
│  ├─ 블록 선택
│  ├─ 블록 분할
│  └─ 블록 병합
└─ 4. 단편화와 효율
   ├─ 내부 단편화
   └─ 외부 단편화
```

## 1. 블록 표현

- **블록 구조**: 힙을 블록 단위로 나누어 관리하는 방식
- **메타데이터(metadata)**: 블록 크기, 할당 여부 등 블록 관리를 위해 저장하는 정보
  - 대표적으로 헤더, 푸터, 정렬 정보, 이전/다음 포인터 등이 있다.

## 2. 빈 공간 추적

- **암시적 가용 리스트(implicit free list)**: 전체 블록을 순차적으로 탐색하면서 가용 블록을 찾는 방식
- **명시적 가용 리스트(explicit free list)**: 가용 블록만 별도의 연결 리스트로 관리하는 방식
  - 가용 상태인 블록은 payload 영역을 사용하지 않으므로, 그 공간에 이전/다음 포인터 같은 관리 정보를 저장할 수 있다.
  - 할당된 블록을 건너뛰고 가용 블록만 탐색하므로 탐색 비용을 줄일 수 있다.
- **분리 가용 리스트(segregated free list)**:
  - 가용 블록을 크기 범위별로 나누어 여러 개의 free list에 관리하는 방식이다.
  - 요청 크기에 맞는 크기 범위의 리스트부터 탐색하므로, 너무 작은 가용 블록을 반복해서 검사하는 비용을 줄일 수 있다.

## 3. 할당/해제 정책

- **블록 선택**: 요청을 처리할 때 어떤 가용 블록을 사용할 것인가
  - **최초 적합(first fit)**: 처음 만나는 충분히 큰 가용 블록을 선택한다.
    - 속도: 중간 / 메모리 효율: 중간
    - 앞부분에서 탐색이 반복되어 비효율이 생길 수 있다.
  - **다음 적합(next fit)**: 이전 탐색이 끝난 위치부터 이어서 탐색한다.
    - 속도: 좋음 / 메모리 효율: 나쁨
    - 같은 앞부분을 반복해서 탐색하는 비용을 줄일 수 있지만, 적당한 블록을 지나쳐 공간 효율이 나빠질 수 있다.
  - **최적 적합(best fit)**: 요청 크기에 가장 가까운 가용 블록을 선택한다.
    - 속도: 나쁨 / 메모리 효율: 좋음
    - 남는 공간을 줄이는 데 유리하지만, 더 좋은 후보를 찾기 위해 탐색 비용이 커질 수 있다.
- **블록 분할**: 할당 후 남는 공간을 별도의 가용 블록으로 나눌 것인가
- **블록 병합**: 해제할 때 인접한 가용 블록을 하나로 합칠 것인가
  - **즉시 병합(immediate coalescing)**: 해제 시점에 바로 병합한다.
  - **지연 병합(deferred coalescing)**: 병합을 미루고, 나중에 필요할 때 병합한다.

## 4. 단편화와 효율

블록 구조와 할당 정책에 따라 메모리 효율이 달라지고, 그 결과 단편화가 발생할 수 있다.

- **내부 단편화(internal fragmentation)**: 할당된 블록 내부에 실제로 사용되지 않고 남는 공간
- **외부 단편화(external fragmentation)**: 가용 공간의 총량은 충분하지만, 여러 곳에 흩어져 있어 큰 연속 공간을 만들기 어려운 상태

# 그 외

* 실제 메모리 할당기는 이보다 훨씬 복잡하다. 동시성, 캐시 친화성, 시스템 호출 비용, 최적화 전략 등 고려해야 할 요소가 많기 때문이다.
* 실제 구현에서는 블록마다 푸터를 두지 않는 방식도 널리 사용된다.
* 이 글에서는 가상 메모리 위의 추상화된 힙 환경을 기준으로 설명한다. 그 아래 계층은 운영체제가 추상화해 제공하므로 여기서는 다루지 않는다.
* `유저 코드 - 표준 라이브러리(malloc) - 시스템 호출 인터페이스(mmap, brk) - 커널/운영체제`처럼 수준을 나누어 생각하면 더 명확하다.
