# 배경 개념

- **세그먼트(segment)**: 주소 공간 중에서 특정 역할을 하는 영역. 대표적으로 스택, 힙, 데이터, 코드 영역이 있다.
- **힙 영역(heap segment)**: 프로그래머가 비교적 직접 제어할 수 있는 큰 메모리 영역이다.
- **힙 할당기(heap allocator)**: 힙 영역의 메모리를 할당하고 해제하는 작은 프로그램이다.
  - 왜 필요한가?: 추상화와 표준화를 제공하고, 여러 환경이나 운영체제에서도 일관되게 동작할 수 있게 해준다.

# 힙 메모리 관리(Heap Memory Management)

동적 메모리 할당은 실행 시간에 변하는 메모리 수요를 처리할 수 있게 해준다. 메모리를 할당하고 회수하면서 변화하는 수요를 효율적으로 감당하는 것이 목적이다.

1. **블록 표현(Representation)**: 힙 메모리를 어떤 블록 단위로 나누고, 각 블록에 어떤 정보를 둘 것인가
2. **빈 공간 추적(Tracking)**: 사용 가능한 가용 블록을 어떤 방식으로 관리할 것인가
3. **할당/해제 정책(Policy)**: 어떤 블록을 선택하고, 어떻게 분할하고, 언제 병합할 것인가

```text
힙 메모리 관리
├─ 1. 블록 표현
│  ├─ 블록 구조
│  └─ 메타데이터
├─ 2. 빈 공간 추적
│  ├─ 암시적 가용 리스트
│  └─ 명시적 가용 리스트
├─ 3. 할당/해제 정책
│  ├─ 블록 선택
│  ├─ 블록 분할
│  └─ 블록 병합
└─ 4. 단편화와 효율
   ├─ 내부 단편화
   └─ 외부 단편화
```

## 1. 블록 표현

- **블록 구조**: 힙을 블록 단위로 나누는 방식
- **메타데이터(metadata)**: 블록 크기, 할당 비트 등 관리 정보
  - 그 외에도 헤더, 푸터, 정렬, 이전/다음 포인터 등이 있다.

## 2. 빈 공간 추적

- **암시적 가용 리스트(implicit free list)**: 전체 블록을 순차적으로 보며 가용 블록을 찾는 방식
- **명시적 가용 리스트(explicit free list)**: 가용 블록만 따로 연결해 관리하는 방식
  - 이것이 가능한 이유는 가용 상태인 블록에서는 페이로드 부분을 관리 정보 저장에 활용할 수 있기 때문이다.
  - 연결 리스트처럼 이전/다음 포인터를 저장해 두면 탐색 횟수를 더 줄일 수 있다.

## 3. 할당/해제 정책

- **블록 선택**: 어떤 가용 블록을 사용할 것인가
  - **최초 적합(first fit)**: 처음 만나는 충분히 큰 가용 블록을 선택
    - 속도: 중간 / 메모리 효율: 중간
    - 앞부분에서 탐색이 반복되어 비효율이 생길 수 있다.
  - **다음 적합(next fit)**: 이전 탐색 위치부터 계속 탐색
    - 속도: 좋음 / 메모리 효율: 나쁨
  - **최적 적합(best fit)**: 요청 크기에 가장 잘 맞는 블록을 선택
    - 속도: 나쁨 / 메모리 효율: 좋음
- **블록 분할**: 할당할 때 남는 공간을 별도 블록으로 나눌 것인가
- **블록 병합**: 해제할 때 인접한 가용 블록을 합칠 것인가
  - **즉시 병합(immediate coalescing)**: 해제될 때 바로 병합
  - **지연 병합(deferred coalescing)**: 나중에 필요할 때 병합

## 4. 단편화와 효율

위의 구조와 정책에 따라 메모리 효율이 달라지고, 그 결과 단편화가 발생한다.

- **내부 단편화(internal fragmentation)**: 할당된 블록 내부에 사용되지 않고 남는 공간
- **외부 단편화(external fragmentation)**: 가용 공간이 흩어져 있어 충분한 총량이 있어도 큰 연속 공간을 만들기 어려운 상태

# 그 외

* 실제 메모리 할당은 이것보다 훨씬 복잡하다. 동시성, 캐시 친화성, 시스템 호출 비용, 최적화 전략 등 고려해야 할 요소가 많기 때문이다.
* 리눅스의 경우 블록에 푸터를 두지 않는 방식도 널리 사용된다.
* 이 글에서는 가상 메모리 위의 추상화된 환경을 기준으로 설명한다. 그 아래 계층은 운영체제가 추상화해 제공하는 영역이므로 여기서는 다루지 않는다.
* `유저 코드 - 표준 라이브러리(malloc) - 시스템 호출 인터페이스(mmap, brk) - 커널/운영체제`처럼 수준을 나누어 생각하면 더 명확하다. 이것들은 가상 메모리와는 별개의 개념이다.
