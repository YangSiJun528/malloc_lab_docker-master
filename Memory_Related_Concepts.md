# 배경 개념

- **자원(resource)**: 프로그램이 실행되기 위해 필요한 하드웨어 또는 운영체제 관리 대상이다. 대표적으로 CPU, 메모리, 파일, 장치가 있다.
- **자원 가상화(resource virtualization)**: 실제 하드웨어 자원을 프로그램에 직접 노출하지 않고, 운영체제가 추상화된 형태로 제공하는 방식이다.
- **커널(kernel)**: 하드웨어 자원에 대한 권한을 가지고, 사용자 프로그램의 요청을 검사하고 수행하는 운영체제의 핵심 부분이다.
- **보호(protection)**: 한 프로그램이나 장치가 다른 프로그램 또는 커널의 자원을 침범하지 못하도록 접근 범위를 제한하는 원리이다.

# 자원 가상화(Resource Virtualization)

자원 가상화는 운영체제가 메모리와 I/O 장치 같은 하드웨어 자원을 직접 드러내지 않고, 프로그램이 안전하고 일관된 방식으로 사용할 수 있도록 추상화하는 방식이다. 프로그램은 자신만의 메모리 공간과 자원을 사용하는 것처럼 보지만, 실제 위치, 할당 시점, 접근 권한은 운영체제와 하드웨어가 관리한다.

1. **주소 공간 가상화(Address Space Virtualization)**: 프로세스마다 독립적인 가상 주소 공간을 제공하고, 이를 물리 메모리에 매핑하는 방식
2. **지연 할당과 페이지 기반 관리(Lazy Allocation & Page-Based Management)**: 실제 자원 할당을 필요한 시점까지 미루고, 페이지 단위로 메모리를 관리하는 방식
3. **보호된 접근과 커널 중재(Protected Access & Kernel Mediation)**: 사용자 프로그램과 장치의 위험한 자원 접근을 커널과 하드웨어가 통제하는 방식

```text
자원 가상화
├─ 1. 주소 공간 가상화
│  ├─ 가상메모리
│  └─ 페이징
├─ 2. 지연 할당과 페이지 기반 관리
│  ├─ demand-zero memory
│  ├─ 페이지 폴트
│  └─ 물리 페이지 할당
└─ 3. 보호된 접근과 커널 중재
   ├─ 시스템콜
   ├─ DMA
   └─ 접근 권한 통제
```

## 1. 주소 공간 가상화

- **가상메모리(virtual memory)**: 각 프로세스가 자신만의 독립적인 메모리 공간을 가진 것처럼 보이게 하는 추상화
- **페이징(paging)**: 가상 주소 공간과 물리 메모리를 고정 크기의 페이지 단위로 나누어 매핑하는 방식

주소 공간 가상화의 핵심은 프로그램이 실제 물리 메모리 주소를 직접 다루지 않는다는 점이다. 프로그램은 가상 주소를 사용하고, 운영체제와 MMU가 가상 주소를 물리 주소로 변환한다.

이 구조 덕분에 각 프로세스는 서로 분리된 주소 공간을 가지며, 한 프로세스가 다른 프로세스의 메모리를 함부로 읽거나 쓸 수 없다. 또한 가상 주소 공간은 연속적으로 보일 수 있지만, 실제 물리 메모리에서는 여러 위치에 흩어져 배치될 수 있다.

## 2. 지연 할당과 페이지 기반 관리

- **demand-zero memory**: 메모리 영역을 요청받았을 때 즉시 물리 페이지를 할당하지 않고, 처음 접근할 때 0으로 초기화된 페이지를 할당하는 방식
- **페이지 폴트(page fault)**: 접근한 가상 페이지가 아직 물리 메모리에 매핑되어 있지 않을 때 발생하는 예외
- **물리 페이지 할당(physical page allocation)**: 페이지 폴트 처리 과정에서 실제 물리 메모리 페이지를 준비해 가상 주소에 연결하는 작업

지연 할당은 자원을 요청받은 순간이 아니라 실제로 사용되는 순간에 할당하는 방식이다. 프로그램이 큰 메모리 영역을 요청하더라도, 운영체제는 모든 물리 페이지를 즉시 준비하지 않을 수 있다.

demand-zero memory가 대표적인 예이다. 프로그램이 새 메모리 영역에 처음 접근하면 페이지 폴트가 발생하고, 커널은 그때 0으로 초기화된 물리 페이지를 할당해 가상 주소에 매핑한다.

이 방식은 페이징을 기반으로 동작한다. 메모리를 페이지 단위로 나누어 관리하기 때문에, 실제로 접근한 페이지만 할당할 수 있다. 결과적으로 운영체제는 사용되지 않는 메모리에 물리 자원을 낭비하지 않고, 필요한 부분에만 자원을 제공할 수 있다.

## 3. 보호된 접근과 커널 중재

- **시스템콜(system call)**: 사용자 프로그램이 커널 기능을 요청하기 위해 사용하는 공식 인터페이스
- **DMA(Direct Memory Access)**: 장치가 CPU를 거치지 않고 메모리에 직접 데이터를 읽거나 쓰는 방식
- **접근 권한 통제(access control)**: 사용자 프로그램이나 장치가 접근할 수 있는 자원의 범위를 제한하는 관리 방식

보호된 접근과 커널 중재의 핵심은 프로그램이나 장치가 하드웨어 자원에 직접 무제한으로 접근하지 못하게 하는 것이다.

사용자 프로그램은 파일, 메모리, 장치 같은 중요한 자원을 직접 조작할 수 없다. 대신 시스템콜을 통해 커널에 요청해야 한다. 커널은 요청이 유효한지, 권한이 있는지, 현재 상태에서 수행 가능한지를 검사한 뒤 실제 작업을 수행한다.

DMA는 장치가 메모리에 직접 접근하므로 성능 면에서 효율적이다. 하지만 장치의 접근 범위를 제한하지 않으면 다른 프로세스나 커널 메모리를 침범할 수 있다. 따라서 DMA 역시 운영체제와 하드웨어의 통제 대상이며, 필요한 경우 IOMMU 같은 기능을 통해 장치의 메모리 접근도 제한하고 가상화한다.

### 1. `mmap()`와 동적 라이브러리 관리

`mmap()`은 파일이나 장치의 내용을 프로세스의 가상 주소 공간 일부에 매핑하는 시스템콜이다. 파일 내용을 직접 `read()`로 복사해서 가져오는 것이 아니라, 특정 가상 주소 범위가 파일의 특정 구간을 가리키도록 연결한다.

동적 라이브러리(`.so`)도 이 방식을 활용한다. 라이브러리 파일은 실행 코드가 들어 있는 `.text` 영역과 전역 변수처럼 수정 가능한 데이터가 들어 있는 `.data` 영역을 가진다. 프로그램이 실행될 때 동적 링커는 이 영역들을 프로세스의 주소 공간에 매핑한다. 보통 `.text`는 읽기/실행 권한으로, `.data`는 읽기/쓰기 권한으로 매핑된다.

이때 주소 공간 가상화 덕분에 여러 프로세스가 같은 라이브러리를 사용하더라도 각자 자기 주소 공간에 라이브러리가 올라온 것처럼 볼 수 있다. `.text`는 수정되지 않는 코드이므로 물리 메모리의 같은 페이지를 여러 프로세스가 공유할 수 있다. 반면 `.data`처럼 수정 가능한 영역은 프로세스마다 다른 상태를 가져야 하므로 private mapping으로 다룬다. 처음에는 같은 내용을 공유하더라도, 어떤 프로세스가 값을 바꾸면 copy-on-write에 의해 그 프로세스만 별도 물리 페이지를 갖게 된다.

또한 라이브러리 전체를 즉시 물리 메모리에 올릴 필요도 없다. 처음 접근한 페이지에서 페이지 폴트가 발생하면, 커널이 그때 필요한 부분만 파일에서 읽어 물리 페이지에 연결한다. 따라서 `mmap()` 기반 동적 라이브러리 관리는 주소 공간 가상화, 페이징, 지연 로딩을 함께 활용하는 예라고 볼 수 있다.

### 2. cow(copy-on-write) 와 fork()

`fork()`는 부모 프로세스와 거의 같은 주소 공간을 가진 자식 프로세스를 만든다. 하지만 힙, 스택, 데이터 영역의 물리 페이지를 전부 즉시 복사하지는 않는다. 커널은 부모와 자식의 페이지 테이블이 같은 물리 페이지를 가리키게 하고, 공유 중인 페이지를 copy-on-write 대상으로 둔다.

둘 다 읽기만 하면 같은 물리 페이지를 계속 공유한다. 어느 한쪽이 공유 페이지에 쓰려고 하면 페이지 폴트가 발생하고, 커널은 그 페이지 하나를 새 물리 페이지로 복사한 뒤 쓰기를 시도한 프로세스의 페이지 테이블만 새 페이지를 가리키도록 바꾼다. 그래서 변경 내용은 해당 프로세스에게만 보이고, 다른 프로세스는 기존 페이지를 계속 본다.

이 방식 덕분에 `fork()`는 큰 프로세스에서도 비교적 빠르게 동작할 수 있다. 특히 `fork()` 직후 `exec()`로 다른 프로그램을 실행하는 경우, 곧 버려질 주소 공간 전체를 미리 복사하지 않아도 된다. COW는 결국 "복사는 필요해질 때까지 미루고, 실제로 수정된 페이지 단위로만 수행한다"는 지연 할당의 한 예이다.

### 3. DMA, 마이크로프로세서, 드라이버

DMA는 CPU가 `load`/`store` 명령으로 데이터를 직접 옮기는 대신, 장치 쪽 하드웨어가 메모리를 직접 읽고 쓰게 하는 방식이다. 이 일을 하는 하드웨어를 보통 DMA controller 또는 DMA engine이라고 부른다.

마이크로프로세서는 명령어를 실행하는 CPU를 하나의 칩으로 구현한 것이다. 보통 코드가 내부에 하드코딩되어 있는 것이 아니라, 메모리나 저장장치의 명령어를 가져와 실행한다. 마이크로프로세서 자체만으로 컴퓨터가 되는 것은 아니고, 메모리와 I/O 장치 같은 주변 구성이 함께 필요하다.

그래서 "DMA가 항상 마이크로프로세서다"라고 말하면 정확하지 않다. 어떤 DMA 엔진은 descriptor를 읽고 주소와 길이를 증가시키며 데이터를 옮기는 전용 상태 기계나 제어 회로에 가깝다. 어떤 고성능 장치는 내부에 펌웨어를 실행하는 작은 프로세서를 함께 가질 수 있지만, 그 경우에도 DMA의 핵심 역할은 범용 프로그램 실행이 아니라 메모리와 장치 사이의 데이터 이동이다.

현대 시스템에서는 네트워크 카드, NVMe SSD, GPU 같은 장치가 자기 안에 DMA 엔진을 가지고 있는 경우가 흔하다. 이런 장치는 버스의 주체가 되어 시스템 메모리에 직접 읽기/쓰기 요청을 보낼 수 있다.

드라이버는 DMA 엔진과 다르다. 드라이버는 운영체제 안에서 실행되는 소프트웨어이고, DMA 엔진은 장치 안에 있는 하드웨어이다. 드라이버는 DMA에 사용할 메모리 버퍼를 준비하고, 장치가 접근할 수 있는 주소를 설정하고, 장치의 레지스터나 descriptor queue에 "어느 주소에서 얼마만큼 전송하라"는 정보를 써 넣는다. 그 다음 실제 데이터 이동은 장치의 DMA 엔진이 수행한다.

장치가 메모리에 직접 접근할 수 있다는 것은 성능상 유리하지만 위험하기도 하다. 잘못 설정된 장치가 아무 메모리나 읽고 쓰면 다른 프로세스나 커널 메모리를 침범할 수 있다. 그래서 운영체제는 IOMMU를 사용해 장치가 접근할 수 있는 메모리 범위를 제한하고, 드라이버는 운영체제의 DMA API를 통해 안전한 주소와 버퍼를 설정한다.

### 4. 예외와 인터럽트 처리 흐름

(페이지 폴트도 아래와 같은 흐름을 따른다.)

CPU는 잘못된 명령 실행, 0으로 나누기, 페이지 폴트, 장치 인터럽트 같은 상황이 생기면 현재 실행 흐름을 잠시 멈추고 예외(exception)나 인터럽트(interrupt)를 발생시킨다. 소프트웨어가 의도적으로 커널에 진입할 때도 `syscall`, `int`, `svc` 같은 명령어로 비슷한 흐름을 사용한다.

이때 CPU는 현재 명령어 위치와 상태 레지스터 같은 최소한의 실행 상태를 저장하고, CPU spec이 정한 방식으로 예외 처리 진입점으로 이동한다. 예를 들어 x86에서는 OS가 IDTR 레지스터에 IDT(Interrupt Descriptor Table)의 위치를 등록하고, ARM 계열에서는 VBAR 같은 레지스터에 vector table의 기준 주소를 등록한다. 예외나 인터럽트가 발생하면 CPU는 예외 번호를 이용해 이 테이블에서 커널 핸들러의 위치를 찾고, 권한 모드를 바꾼 뒤 그 주소로 점프한다.

핸들러는 원인을 확인하고 필요한 처리를 한다. 예를 들어 페이지 폴트라면 접근이 유효한지 검사한 뒤 물리 페이지를 할당하거나, 잘못된 접근이면 프로세스를 종료할 수 있다. 처리가 끝나면 저장해 둔 컨텍스트를 복구하고 원래 프로그램으로 돌아가거나, 더 이상 실행할 수 없는 경우 에러를 보고하고 종료한다.
